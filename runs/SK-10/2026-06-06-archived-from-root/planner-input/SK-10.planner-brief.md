# Planner Brief — SK-10: As a valid user I want to log in successfully

## Story summary

A standard user signs in to Saucedemo with valid credentials and lands on
the inventory page, which shows the product list and no error message. This
is a happy-path-only login story; negative cases (invalid password, locked
user) are out of scope and covered by sibling stories SK-11 / SK-12.

## Acceptance criteria

0. standard_user can log in with valid credentials
1. user lands on inventory page
2. product list is visible
3. no error message is shown

## Risks (anchors)

- **RISK-001** (high): a valid user cannot log in at all — blocks all access.
  Related ACs: [0, 1].
- **RISK-002** (medium): login "succeeds" but the inventory page is not
  usable / product-less. Related ACs: [1, 2].
- **RISK-003** (low): a spurious error is shown on a successful login.
  Related ACs: [3].

## In-scope scenarios for the Playwright Planner

- **Valid login reaches inventory** (TC-001, RISK-001/002): navigate to the
  login page, enter `standard_user` / `secret_sauce`, submit, assert the URL
  moves to the inventory page and the login form is gone.
- **Inventory shows the product list** (TC-002, RISK-002): on the post-login
  page, assert at least one product item is visible.
- **No error on success** (TC-003, RISK-003): after a valid login, assert no
  error banner / error text is present.

All three can live in one spec/test file for SK-10; they share the same
login action and differ in the assertion.

## Out-of-scope for the Planner

- Invalid-password / wrong-credentials behaviour (→ SK-11).
- Locked-out user behaviour (→ SK-12).
- Empty-field / form-validation behaviour (not in this story's ACs).
- Logout (→ SK-18), cart, checkout, sorting (other stories).
- Visual styling, performance/latency of login.
- Any API-level assertion: this demo target (Saucedemo) exposes no real
  backend API for login, so everything stays E2E.

## UI baseline notes

- Saucedemo login: username `[data-test="username"]`, password
  `[data-test="password"]`, login button `[data-test="login-button"]`
  (stable test hooks — prefer these per the locator policy).
- Successful login navigates to `/inventory.html`; products render as
  `[data-test="inventory-item"]` (confirm live via the playwright-test MCP
  during planning — do not assume; §3.8 "don't generate from text alone").

## Ambiguities still open

- Scope is happy-path only (confirmed at Gate 1, non-blocking). The Planner
  must NOT add negative-login scenarios — those are other stories.
