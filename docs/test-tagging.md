# Test Tagging — `@smoke` / `@regression`

> **Status:** continuous improvement. A convention for tagging the **generated
> product tests** (the Playwright E2E specs in `tests/`, and conceptually the
> Newman API checks) so a fast subset can run as a smoke gate and the full set
> as regression. This is about the **app under test**, not the pipeline's own
> tests — those live in `test/` and gate `main` via `quality-checks`
> (`docs/pipeline-architecture.md` §6.1, "two test layers").

---

## 1. The two tags

| Tag           | Means                                                                                          | Runs          |
| ------------- | ---------------------------------------------------------------------------------------------- | ------------- |
| `@smoke`      | A few critical "is it alive / does the core path work" checks. Fast. Run on every PR / deploy. | Subset        |
| `@regression` | The full safety net — every business-critical case, including edge/negative cases.             | All / nightly |

A test can carry **both** (`['@smoke', '@regression']`) — a core login test is
both a smoke check and part of regression. A narrow edge case is usually
`@regression` only. Untagged tests still run under a plain `playwright test`;
the tags add _filtered subsets_, they don't hide anything.

This maps directly onto the Automation Decision Model
(`docs/automation-decision-model.md`): a case the Test Designer marks
"smoke / regression critical" → `automate_e2e` is exactly what should carry
`@smoke`.

---

## 2. How to tag (Playwright)

Use the test-options form (Playwright 1.42+; this repo is on 1.60):

```ts
test(
  'user logs in and lands on inventory',
  { tag: ['@smoke', '@regression'] },
  async ({ page }) => {
    // ...
  }
);
```

A whole group can be tagged on the `describe`:

```ts
test.describe('Checkout', { tag: ['@regression'] }, () => {
  /* ... */
});
```

The seed test (`tests/seed.spec.ts`) is tagged `@smoke` — "the app loads" is the
canonical smoke check and a worked example of the convention.

---

## 3. Running a subset

```bash
npm run test:e2e:smoke        # playwright test --grep @smoke
npm run test:e2e:regression   # playwright test --grep @regression
npm test                      # everything (tagged or not)
```

`--grep` matches the tag in the test title/options; `--grep-invert @smoke` runs
everything _except_ smoke. (API branch: Newman has no native tag filter — keep
smoke-vs-regression API checks in separate collections, or name requests with a
`[smoke]` prefix and filter by folder; documented here so the convention is
consistent across both branches.)

---

## 4. Who applies the tags

- **The Test Designer** decides smoke-vs-regression criticality per case (it
  already records this reasoning in `automation_decision_reason`). That decision
  is the input to the tag.
- **The Playwright Generator** (native agent) writes the test; the generated
  test SHOULD carry the tag implied by the case's criticality. Because the
  native agents are regenerated by `init-agents` (not hand-edited), the
  convention lives here + in the planner brief / seed example so the Generator
  has it as context, rather than in the agent file itself.
- **Gate 4** is where a human confirms the tag is right along with the rest of
  the test (a "smoke" test that's actually slow/flaky, or a critical path left
  untagged, is a Gate-4 finding).

---

## 5. Gating posture (the decision)

Today the generated E2E suite runs in CI as **`playwright-full` —
informational** (`continue-on-error`), by design: a generated test can be
legitimately red mid-development without blocking an unrelated PR
(`docs/pipeline-architecture.md` §6.1).

**Graduation path** (recommended, when the suite is stable):

1. Keep `playwright-full` (full suite) **informational**.
2. Add a **`playwright-smoke`** job that runs `npm run test:e2e:smoke` and make
   **that** a required check. A broken core path _should_ block; a flaky edge
   case should not.
3. Promote more tags into smoke as confidence grows; promote the full suite to
   blocking only once its pass rate is consistently green (the metrics +
   `/evolve` loop tells you when).

This staged approach gets a real merge-gating smoke signal without the
all-or-nothing risk of making a large generated suite blocking on day one. It is
**not enabled yet** — it's the documented next step, gated on the suite proving
stable across runs.
